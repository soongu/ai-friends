# Day 1: 서버 to 서버(S2S)의 기초 - AI를 우리 앱에 불러오자

## 🎯 학습 목표
- 학생들이 **서버 to 서버(S2S) 통신의 핵심 개념**을 초보자 눈높이에서 이해하고, 왜 백엔드 개발에서 필수인지 감을 잡습니다.
- 이번 프로젝트 과제인 **포트원 결제 API 연동**을 미리 연결해, "왜 지금 HTTP 클라이언트를 배우는지"를 명확히 납득시킵니다.
- 유지보수가 중단된 `RestTemplate` 대신, 최신 Spring Boot 환경에 맞는 **동기식 HTTP 클라이언트 `RestClient`** 사용법을 익힙니다.
- 비동기식 클라이언트인 **`WebClient`의 역할과 사용 시점**을 가볍게 소개해 기술 선택의 기준을 세웁니다.
- Spring AI 없이 **Google Gemini API를 직접 호출**하는 프록시 API를 구현하며, 실무형 외부 API 연동 흐름을 체험합니다.

## 🛠️ 브랜치 준비 작업 (Pre-work)
*`day1` 브랜치 상태*
1. **기본 프로젝트 기동 확인**: 스프링 부트 서버가 정상 실행되고, 기본 컨트롤러 응답이 확인되는 상태입니다.
2. **환경 변수 준비**: `GEMINI_API_KEY`를 로컬 환경 변수 또는 `.env`로 미리 준비합니다.
3. **HTTP 클라이언트 학습 전제**: 학생들에게 "외부 API 호출은 결국 백엔드의 HTTP 요청"이라는 관점을 먼저 심어둡니다.
4. **실습 API 테스트 도구 준비**: Postman(또는 IntelliJ HTTP Client)을 사용할 수 있게 안내합니다.
5. **실습 시나리오 공유**: 프론트는 프롬프트만 보내고, 백엔드가 Gemini 요청 JSON을 조합해 대신 호출하는 프록시 흐름을 미리 예고합니다.

## ⏱️ 3시간(180분) 라이브 코딩 타임테이블 (Minute-by-Minute)

### 00:00 ~ 00:30 (30분) - Part 1: S2S 개념을 감으로 이해하기
*대상: 개념 설명 + 간단한 요청/응답 흐름 다이어그램*

- **(0~10분) "브라우저가 아니라 서버가 서버를 부른다"**:
  - S2S를 일상 비유로 설명합니다. "우리 앱 백엔드는 고객센터 직원, 외부 API는 타 부서 담당자"처럼 역할로 풀어 설명합니다.
  - 클라이언트-서버와 서버-서버의 차이를 요청 주체 중심으로 비교합니다.

- **(10~20분) 포트원 결제 API 과제와 연결**:
  - "결제 승인/조회는 프론트가 아니라 백엔드가 포트원 서버와 통신해야 안전하다"는 보안 이유를 짚습니다.
  - API 키 보호, 결제 위변조 방지, 서버 검증의 필요성을 강조합니다.

- **(20~30분) 요청/응답 데이터 흐름 스케치**:
  - `Client -> Our Server -> External API -> Our Server -> Client` 흐름을 화이트보드로 정리합니다.
  - "우리 서버는 단순 전달자가 아니라 요청 가공/검증/로깅/에러 처리 책임이 있다"는 메시지를 심어줍니다.

### 00:30 ~ 01:15 (45분) - Part 2: RestClient로 동기식 API 호출 구현
*대상 파일: `build.gradle`, `config/HttpClientConfig.java`, `service/GeminiService.java` (또는 동등 구조)*

- **(30~40분) 왜 `RestTemplate`이 아닌가?**:
  - `RestTemplate`은 신규 기능 개발이 사실상 멈춘 상태라는 점을 짚고, 최신 권장 흐름을 안내합니다.
  - 동기식 요구사항에는 `RestClient`가 단순하고 읽기 좋다는 장점을 설명합니다.

- **(40~55분) `RestClient` 기본 세팅**:
  - `RestClient.builder()` 기반 빈 등록을 시연합니다.
  - 기본 URL, 헤더, 타임아웃, 공통 예외 처리 포인트를 함께 설명합니다.

- **(55~75분) 외부 API 호출 메서드 작성**:
  - 서비스 레이어에서 요청 DTO를 JSON으로 조합해 `POST` 호출을 구현합니다.
  - 응답 본문 파싱 후 우리 서비스 응답 DTO로 변환합니다.
  - 실패 케이스(4xx/5xx) 로그와 사용자용 메시지 분리 전략을 보여줍니다.

### 01:15 ~ 01:25 (10분) - 휴식 (Break)

### 01:25 ~ 01:40 (15분) - Part 3: WebClient는 언제 쓰는가? (가벼운 소개)
*대상: 개념 비교 중심 (구현 최소화)*

- **(85~95분) `WebClient` 핵심만 소개**:
  - WebFlux에서 제공하는 비동기/논블로킹 클라이언트임을 설명합니다.
  - 대량 I/O, 높은 동시성, 스트리밍 같은 요구가 있을 때 강점이 있음을 예시로 전달합니다.

- **(95~100분) 오늘은 왜 `RestClient` 중심인지 정리**:
  - 현재 과제 범위는 복잡한 리액티브 파이프라인보다 "명확한 요청/응답 흐름"이 중요함을 강조합니다.
  - "지금은 이해와 생산성, 이후 규모가 커지면 WebClient 확장"이라는 학습 로드맵을 제시합니다.

### 01:40 ~ 02:45 (65분) - Part 4: Gemini API 프록시 라이브 코딩 (Spring AI 미사용)
*대상 파일: `controller/AiController.java`, `service/GeminiService.java`, `dto/*`*

- **(100~115분) 프록시 API 스펙 정의**:
  - 클라이언트가 보낼 최소 입력(`prompt`)과 서버가 반환할 출력(`answer`, `model`, `usage` 등)을 정의합니다.
  - "프론트는 단순하게, 백엔드는 똑똑하게" 원칙을 강조합니다.

- **(115~130분) 요청 JSON 조합 로직 구현**:
  - Gemini 요구 포맷에 맞는 JSON 구조(`contents`, `parts`)를 백엔드에서 조합합니다.
  - 프롬프트 전처리(빈 값 검증, 길이 제한, 금칙어/안전장치 기본 틀)를 추가합니다.

- **(130~150분) Gemini 서버 호출 + 응답 가공**:
  - `RestClient`로 구글 서버에 요청을 보내고, 응답에서 핵심 텍스트를 추출합니다.
  - 실패 응답 시 상태코드별 처리와 디버깅 로그 전략을 시연합니다.

- **(150~165분) 컨트롤러 완성 및 통합 테스트**:
  - `/api/ai/ask` 같은 엔드포인트를 완성하고 Postman으로 실시간 검증합니다.
  - 요청/응답 로그를 보며 S2S 왕복 과정을 학생들과 함께 추적합니다.

### 02:45 ~ 03:00 (15분) - 마무리
- **(165~180분) Day 1 회고 + 프로젝트 과제 연결**:
  - 오늘 핵심 복습: S2S 개념 -> RestClient 실습 -> WebClient 포지셔닝 -> Gemini 프록시 구현.
  - "이제 포트원 결제 API도 같은 패턴으로 붙일 수 있다"는 자신감을 심어줍니다.
  - Day 2 예고: 인증/인가 또는 결제 플로우에 필요한 도메인 설계 심화.

## 💡 강사 팁 (Instructor Tips)
- **초급자에게는 용어보다 역할**: "누가 누구에게 왜 요청하는가"를 먼저 설명하고, 그 다음에 `RestClient`, `WebClient` 용어를 붙이세요.
- **S2S를 과제와 즉시 연결**: 포트원 결제를 예시로 계속 연결하면 학생들이 기술의 쓸모를 바로 이해합니다.
- **WebClient는 욕심내지 않기**: 오늘은 비동기 코드 디테일보다 "언제 선택하는가"만 명확히 남기는 것이 좋습니다.
- **프록시 패턴의 실무 포인트 강조**: API 키는 절대 프론트에 두지 않고 백엔드에서만 관리한다는 원칙을 반복하세요.
- **실패 시나리오도 꼭 시연**: 200 응답 데모만 하면 학습이 얕아집니다. 401/429/500 같은 실패 상황 대응을 보여줘야 실전 감각이 생깁니다.
